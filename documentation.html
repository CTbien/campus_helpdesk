Documentation Campus HelpDesk

Guide complet de l'architecture, du code et du fonctionnement du projet.

Qu'est-ce que Campus HelpDesk ?

C'est une application web de gestion de tickets d'assistance pour un campus. Les étudiants créent des tickets pour signaler des problèmes (Wi-Fi, matériel, logiciel…). Les techniciens traitent ces tickets. Les administrateurs supervisent le tout : utilisateurs, catégories et statistiques.

Architecture du projet

Le projet suit une architecture MVC simplifiée en PHP natif (sans framework) :

Les 3 couches

  • Repositories (Modèle) → Accèdent à la base de données via PDO. Chaque table a son Repository.
  • Services (Contrôleur / Logique métier) → Utilisent les Repositories et appliquent les règles métier (validation, hashing, contrôle d'accès).
  • Pages PHP publiques (Vue + Contrôleur frontal) → Chaque fichier dans public/ est à la fois le contrôleur de la page ET la vue HTML.

Le flux d'une requête est toujours :

1
→
Navigateur appelle public/student/dashboard.php
2
→
Page PHP vérifie la session, puis appelle un Service
3
→
Service appelle un Repository pour lire/écrire en BDD
4
→
Repository exécute une requête SQL préparée via PDO
5
→
Les données remontent → la page PHP génère le HTML avec Bootstrap

Configuration BDD — db.php

app/config/db.php

Ce fichier définit une seule fonction db() qui retourne une instance PDO unique (pattern Singleton) :

function db(): PDO {
  static $pdo = null;          // Variable statique = une seule instance
  if ($pdo === null) {
    $dsn = "mysql:host=127.0.0.1;dbname=campus_helpdesk;charset=utf8mb4";
    $pdo = new PDO($dsn, "root", "", [
      PDO::ATTR_ERRMODE => PDO::ERRMODE_EXCEPTION,       // Erreurs = exceptions
      PDO::ATTR_DEFAULT_FETCH_MODE => PDO::FETCH_ASSOC,  // Résultats = tableaux associatifs
      PDO::ATTR_EMULATE_PREPARES => false,                // Vraies requêtes préparées
    ]);
  }
  return $pdo;
}

static $pdo : la variable survit entre les appels. On ne crée la connexion qu'une seule fois par requête HTTP.

Middleware — auth.php

app/middleware/auth.php

Deux fonctions utilitaires de sécurité :

function requireLogin(): void {
  // Redirige vers login si pas connecté
  if (!isset($_SESSION['user'])) {
    header('Location: login.php');
    exit;
  }
}

function requireRole(array $roles): void {
  // Bloque l'accès si le rôle n'est pas autorisé
  $role = $_SESSION['user']['role'] ?? null;
  if (!$role || !in_array($role, $roles, true)) {
    http_response_code(403);
    echo "<h1>403 Interdit</h1>";
    exit;
  }
}

AuthService.php

app/services/AuthService.php — Gère la connexion.

Méthode login($email, $password)

  1. Valide le format de l'email avec filter_var()
  2. Cherche l'utilisateur en BDD via UserRepo::findByEmail()
  3. Vérifie que le compte est actif (actif = 1)
  4. Compare le mot de passe avec le hash via password_verify()
  5. Retourne ['ok' => true, 'user' => [...]] ou ['ok' => false, 'error' => '...']

Méthode redirectByRole($role)

Redirige automatiquement vers le bon dashboard selon le rôle :

  • ADMIN → ../admin/dashboard.php
  • TECH → ../tech/dashboard.php
  • ETUDIANT → ../student/dashboard.php

TicketService.php

app/services/TicketService.php — Couche métier entre les pages et le TicketRepo.

Pourquoi un Service en plus du Repository ?

Le Repository ne fait que du SQL. Le Service ajoute la logique métier : validation des données, gestion des nouvelles catégories, contrôle d'accès (un étudiant ne peut pas voir le ticket d'un autre), etc.

Pages — Authentification

login.php

public/auth/login.php

  1. Si l'utilisateur est déjà connecté (session existante), il est redirigé automatiquement vers son dashboard
  2. Si POST : on appelle AuthService::login(), on stocke l'utilisateur en session, on régénère l'ID de session (anti session fixation), puis on redirige
  3. Affiche un formulaire Bootstrap simple (email + mot de passe)

logout.php

public/auth/logout.php

Détruit complètement la session : vide $_SESSION, supprime le cookie de session, appelle session_destroy(), puis redirige vers login.

Pages — Espace Étudiant

Toutes ces pages vérifient que $_SESSION['user'] existe, sinon → redirection vers login.

dashboard.php

Affiche la liste des tickets de l'étudiant connecté dans un tableau (appelle getStudentTickets()). Chaque ligne est cliquable pour voir le détail. Bouton "Nouveau Ticket" en haut.

create_ticket.php

Formulaire de création avec : titre, catégorie (dropdown + option "Autre"), priorité, description. Sur soumission, appelle TicketService::createTicket().

Fonctionnalité intéressante

Quand l'étudiant choisit "Autre" dans le menu des catégories, un champ texte apparaît (via JavaScript) pour saisir le nom d'une nouvelle catégorie. Le service la crée automatiquement en BDD.

ticket_detail.php

Affiche le détail complet d'un ticket + l'espace d'échange (messages). L'étudiant peut ajouter des messages seulement si le ticket n'est pas résolu. Il ne peut pas changer le statut.

Pages — Espace Technicien

Accès réservé au rôle TECH.

dashboard.php

File d'attente avec filtres dynamiques (statut + priorité, via des <select> avec onchange="this.form.submit()"). Les tickets sont triés : OPEN en premier, puis IN_PROGRESS, puis RESOLVED. Bouton "œil" pour consulter chaque ticket.

ticket_detail.php

Interface complète de traitement. Le technicien peut :

  • S'assigner le ticket (bouton "Assigner à moi") → passe automatiquement en IN_PROGRESS
  • Changer le statut (OPEN / IN_PROGRESS / RESOLVED) via un dropdown
  • Envoyer des messages dans l'espace d'échange

Note : Ce fichier accepte aussi les ADMIN (vérification in_array($role, ['TECH', 'ADMIN'])).

Pages — Espace Admin

Accès réservé au rôle ADMIN.

dashboard.php

Hub central avec 3 cartes cliquables (Utilisateurs, Paramètres, Statistiques) + aperçu des 5 tickets les plus récents (cliquables).

users.php + user_form.php + user_toggle.php

Gestion complète des utilisateurs :

  • users.php : tableau avec tous les utilisateurs, leur rôle (badge coloré), leur statut (Actif/Inactif), et boutons Éditer/Désactiver
  • user_form.php : formulaire unique pour créer OU éditer (détecte le paramètre ?id= dans l'URL). En édition, le mot de passe est optionnel
  • user_toggle.php : action POST qui active/désactive un utilisateur (empêche l'admin de se désactiver lui-même)

tickets.php + ticket_detail.php

Liste complète de tous les tickets avec filtres (comme le tech dashboard) + vue détaillée avec toutes les actions (assigner, changer statut, envoyer messages).

settings.php

Gestion des catégories de tickets : ajouter, renommer (édition inline avec JS), supprimer (avec confirmation). La suppression échoue si des tickets utilisent la catégorie.

statistics.php

Dashboard visuel avec :

  • 4 cartes KPI : Total, Ouverts, En cours, Résolus
  • 3 graphiques Chart.js : Doughnut (par statut), Pie (par priorité), Bar (par catégorie)

1
→
Création : L'étudiant remplit le formulaire → statut = OPEN, assigne_a = NULL
2
→
Prise en charge : Un tech clique "Assigner à moi" → statut = IN_PROGRESS, assigne_a = tech.id
3
→
Échange : Tech et étudiant discutent via les messages du ticket
4
→
Résolution : Le tech clique "Marquer Résolu" → statut = RESOLVED. L'étudiant ne peut plus poster

Sécurité

Mesures de sécurité implémentées

  • Requêtes préparées PDO : toutes les requêtes SQL utilisent des paramètres nommés (:email, :id) → protection contre les injections SQL
  • password_hash / password_verify : les mots de passe sont hashés en bcrypt → jamais stockés en clair
  • htmlspecialchars() : partout dans l'affichage HTML → protection contre les attaques XSS
  • session_regenerate_id(true) : après le login → protection contre le session fixation
  • Contrôle d'accès par rôle : chaque page vérifie le rôle en session. Un étudiant ne peut pas accéder aux pages tech/admin
  • Contrôle côté serveur : un étudiant ne peut voir/modifier que SES tickets (vérifié dans TicketService, pas juste dans le HTML)
  • Destruction complète de session : le logout vide la session ET supprime le cookie
  • declare(strict_types=1) : active le typage strict PHP → réduit les bugs liés aux conversions implicites
